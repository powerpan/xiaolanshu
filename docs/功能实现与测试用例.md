# 功能实现与测试用例

本文档说明跃鹿运动当前饮食推荐、训练打卡、训练复盘和管理端的实现方式，并给出可直接执行的测试用例。文档面向接手仓库的同事，默认已经按 `README.md` 或 `docs/Windows部署操作流程.md` 启动后端和前端。

## 饮食推荐功能实现

饮食推荐的用户入口在前端 `vue-project/src/mainpage.vue` 的“饮食建议”页面。页面加载时调用 `vue-project/src/services/nutritionApi.js` 中的 `getNutritionRecommendation()` 和 `getNutritionLibrary()`，分别读取 `/nutrition/recommendation` 和 `/nutrition/library`。普通用户页面只展示推荐结果，不再展示单独的偏好设置表单。

后端入口集中在 `fitnessGuidance/src/main/java/com/xiaolanshu/fitnessGuidance/controller/NutritionController.java`。`/nutrition/recommendation` 会先解析 `Authorization` 或 `jwttoken` 中的 JWT，拿到当前用户名后读取 `users`、`userprofiles` 和后端兼容保留的 `nutritionpreferences` 数据，再由 `buildRecommendation()` 实时组装 `NutritionRecommendation` 返回给前端。未登录或 token 失效时返回 `未登录`。

核心推荐计算逻辑如下：

- 基础体重来自 `users.weight`，没有体重时使用 65kg 兜底。
- 训练目标来自 `userprofiles.fitness_goal`，没有目标时按“保持健康”兜底。
- 每周训练频率来自 `userprofiles.weekly_frequency`，没有频率时按每周 3 次兜底。
- 目标热量先按 `体重 * (28 + min(训练频率, 6) * 1.2)` 估算；增肌和提升力量增加约 300kcal，减脂和塑形减少约 250kcal，最终不低于 1300kcal。
- 蛋白质按目标调整系数：保持健康约 `1.5g/kg`，增肌和提升力量约 `1.8g/kg`，减脂和塑形约 `1.7g/kg`。
- 脂肪按目标调整系数：保持健康约 `0.8g/kg`，增肌和提升力量约 `0.9g/kg`，减脂和塑形约 `0.7g/kg`。
- 碳水由剩余热量换算，最低保留 100g，避免建议过度低碳。
- 饮水按体重估算，每周训练 4 次及以上按 `38ml/kg`，否则按 `35ml/kg`。

推荐结果包含五类内容：

- 指标类：目标热量、蛋白质、碳水、脂肪、饮水量。
- 总结类：当前目标、宏量营养比例、训练日和休息日提醒。
- 餐次类：早餐、午餐、训练前后、晚餐和可选加餐。
- 执行类：训练前后时机、食物选择、购物清单、风险提醒。
- 模板类：从管理端维护的数据中读取餐单模板、食物替换规则和外食场景建议。

饮食模板库由 `NutritionLibraryMapper` 直接访问数据库，主要表结构在 `fitnessGuidance/src/main/resources/schema.sql`：

- `fooditems`：食材名称、类别、每 100g 热量、蛋白质、碳水、脂肪、标签和适用场景。
- `mealtemplates`：餐单名称、餐次、目标、场景、目标热量、说明和食物组合。
- `foodreplacements`：原食物、替换食物、替换原因和类别。
- `eatingscenarios`：食堂、便利店、外卖、家庭做饭等场景策略。

普通用户读取 `/nutrition/library` 时，后端会根据用户训练目标和饮食场景过滤模板；管理员读取 `/nutrition/admin/library` 时返回完整模板库。前端把推荐内容压缩为“推荐餐单”“食材与替换”“场景与注意”三个可展开区块，避免饮食页过长。

## 训练打卡与复盘功能实现

训练打卡分为普通“今日打卡”和“训练计划内按完成情况打卡”两种入口。普通入口在“训练打卡”页面填写时长、状态和备注；训练计划入口在“训练计划”页面勾选动作后点击“按完成情况打卡”。

前端训练相关接口在 `vue-project/src/services/trainingApi.js`：

- `getSplitMode(daytime)` 调 `/fitnessplan/getsplitmode`，获取第几天的训练模式。
- `getActionTasks(daytime)` 调 `/fitnessplan/getactiontask`，获取动作列表。
- `getPlanTaskRecords(daytime)` 调 `/fitnessplan/taskrecords`，读取当天动作完成记录。
- `savePlanTaskRecord(payload)` 调 `/fitnessplan/taskrecord`，保存单个动作是否完成。
- `saveTodayCheckin(payload)` 调 `/checkin/today`，保存今日打卡。
- `getCheckinStats()` 调 `/checkin/stats`，读取复盘统计。
- `getTrainingPlanInsight()` 调 `/fitnessplan/insight`，读取计划反馈、恢复提示和调整建议。

训练计划当前允许提前查看后续天数。前端第 1 到第 7 天按钮都可以点击并加载计划，但如果前序计划日还没有打卡，后续天数只允许查看，不允许勾选动作或生成打卡。页面会提示“可提前查看第 N 天训练计划；完成第 X 天打卡后，才可打卡第 N 天”。

打卡顺序限制同时由前后端保护：

- 前端通过 `firstLockedPlanDay(day)` 检查当前用户最近打卡记录里的 `planDay`，如果前序天数缺失，禁用动作勾选和“按完成情况打卡”按钮。
- 后端在 `FitnessCheckinServiceImpl.validatePlanDayOrder()` 中再次校验 `fitnesscheckins.plan_day`。如果用户绕过前端直接请求第 2 天及之后的计划打卡，而第 1 天没有记录，后端会返回 `请先完成第 X 天训练打卡，再打卡第 N 天`。
- 同一天已经记录过某个 `planDay` 时，后端不允许把同一天改成另一个计划天数，避免一条打卡记录代表多个训练计划日。

训练计划内勾选动作后，前端调用 `/fitnessplan/taskrecord` 保存到 `plantaskrecords` 表。该表以 `username + plan_date + daytime + action_index` 做唯一键，同一天同一个计划日的同一个动作反复勾选会更新原记录。刷新页面后前端会重新读取后端记录，并同步到本地完成状态。

计划内“按完成情况打卡”会根据已完成动作数生成今日打卡：

- 时长取当前表单时长和计划估算时长中的较大值。
- 状态在全量完成时写“状态不错”，部分完成时写“有挑战”。
- 备注自动写入第几天、训练模式和动作完成比例。
- 请求 `/checkin/today` 时带上 `planDay`，用于后端顺序校验和复盘统计。

训练复盘由 `FitnessCheckinServiceImpl.getStats()` 和 `/fitnessplan/insight` 两部分组成：

- `/checkin/stats` 返回累计打卡天数、累计分钟、平均分钟、本月打卡天数、当前连续、最佳连续和最近记录。
- 当前连续打卡按今天往前倒推，允许今天已打卡时从今天开始连续计算。
- 最佳连续打卡从历史日期顺序扫描，计算最长连续日期段。
- `/fitnessplan/insight` 读取本周动作记录数、完成动作数、完成率、训练周期、恢复日、当前阶段、恢复提示和最近计划调整记录。
- 前端“训练复盘”页面展示准备度、最佳连续、平均时长、最近 35 天热力图、最近记录和教练建议。

## 管理端实现

管理端不是单独应用，而是同一个 Vue 前端根据登录用户身份切换视图。登录后前端读取 `/user/getmessage`，如果 `identity` 为 `ADMIN`，`mainpage.vue` 的导航只展示内容管理、动作库管理、饮食模板和用户管理；普通用户只展示训练、饮食、打卡、复盘和内容阅读入口。

前端管理端主要实现点：

- `navGroups` 根据 `isAdmin` 计算菜单。
- `openView()` 会校验目标页面是否属于当前角色，普通用户访问管理页会提示“该区域仅管理员可见”，管理员访问普通训练页会提示“管理员端仅显示管理功能”。
- 内容、动作、饮食、用户管理表单都放在 `mainpage.vue` 中，通过 `services` 目录下的接口模块与后端通信。
- 文章、公告、动作和用户编辑使用弹窗或独立编辑表单，避免把编辑内容回写到新增表单。

后端管理权限通过各控制器重复校验实现。核心规则是：解析 JWT 获取用户名，再查 `users.identity` 是否等于 `ADMIN`；不满足时返回 `无相关权限`。

管理端功能范围如下：

- 内容管理：`ArticleController` 和 `NoticeController` 提供文章、公告的新增、编辑、删除、详情、草稿读取和草稿清空。普通用户只能读取已发布内容，写操作必须是管理员。
- 动作库管理：`ExerciseGuideController` 提供动作查询、筛选、新增、编辑、删除和图片上传。图片上传只接受图片 MIME 类型，扩展名限制为 jpg、jpeg、png、webp、gif，文件保存到后端 `uploads/exercise-guides`。
- 饮食模板管理：`NutritionController` 的 `/nutrition/admin/*` 接口维护食材、餐单模板、替换规则和外食场景，写操作均要求管理员身份。
- 用户管理：`UserController` 提供全部用户列表、添加用户、编辑用户资料、注册审核和删除用户。审核通过时自动创建默认健身需求，审核不通过或删除用户时同步清理用户资料，删除用户还会删除其文章。

管理端数据写入后，普通用户页面通过重新读取对应接口看到更新结果。例如管理员新增餐单模板后，普通用户饮食建议页刷新会重新读取 `/nutrition/recommendation` 和 `/nutrition/library`，新的餐单模板会出现在推荐餐单区块。

| 用例编号 | 场景 | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- | --- |
| NUT-001 | 未登录访问饮食推荐接口 | 清空浏览器 token 或使用无效 token | 请求 `/nutrition/recommendation` | 返回 `未登录`，前端不展示推荐数据 |
| NUT-002 | 普通用户查看饮食建议 | 使用 `demo / demo123` 登录 | 进入“饮食建议” | 页面展示热量、蛋白质、碳水、脂肪、饮水量和三块可展开内容 |
| NUT-003 | 饮食页不再显示偏好设置 | 普通用户已登录 | 进入“饮食建议”并滚动页面 | 不出现“偏好设置”“保存偏好”“饮食类型”“预算”“外食频率”等表单项 |
| NUT-004 | 健身目标影响推荐 | 普通用户已登录 | 在“健身需求”把目标改为“减脂”并保存，再进入“饮食建议” | 推荐摘要体现减脂目标，热量不低于 1300kcal，提醒包含控制隐形热量或低油策略 |
| NUT-005 | 训练频率影响饮水和热量 | 普通用户已登录 | 把每周频率改为 5 次并保存，再刷新饮食建议 | 推荐中的热量随频率上调，饮水按较高训练频率估算 |
| NUT-006 | 管理端新增餐单模板后用户可见 | 管理员已登录 | 在“饮食模板”新增与目标匹配的餐单模板；普通用户重新进入饮食建议 | 推荐餐单区块出现新增模板内容 |
| NUT-007 | 管理端新增替换规则后用户可见 | 管理员已登录 | 新增一条食物替换规则；普通用户刷新饮食建议 | “食材与替换”区块出现新增替换规则 |
| NUT-008 | 管理端新增外食场景后用户可见 | 管理员已登录 | 新增与目标匹配的外食场景；普通用户刷新饮食建议 | “场景与注意”区块出现新增场景建议 |
| NUT-009 | 普通用户不能访问饮食模板管理 | 普通用户已登录 | 手动尝试进入管理端饮食模板视图或直接请求 `/nutrition/admin/library` | 前端提示无权限或接口返回 `无相关权限` |
| NUT-010 | 模板库为空或过滤结果为空时兜底 | 清空或过滤掉对应目标模板 | 普通用户进入饮食建议 | 核心热量、宏量营养和基础建议仍可展示，不因模板为空导致页面报错 |

| 用例编号 | 场景 | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- | --- |
| CHK-001 | 未登录访问今日打卡 | 清空浏览器 token 或使用无效 token | 请求 `/checkin/today` | 返回 `未登录` |
| CHK-002 | 保存普通今日打卡 | 普通用户已登录 | 进入“训练打卡”，填写时长、状态、备注并保存 | 页面提示保存成功，累计天数、累计分钟和最近记录刷新 |
| CHK-003 | 查看后续训练计划 | 普通用户未完成第 1 天计划打卡 | 进入“训练计划”，点击第 2 天或更后天数 | 能看到对应天数动作列表，页面提示可提前查看但不能提前打卡 |
| CHK-004 | 后续计划不能提前勾选动作 | 普通用户未完成前序计划日 | 打开第 2 天计划，尝试勾选动作 | 动作勾选框禁用，不能保存动作完成记录 |
| CHK-005 | 后续计划不能提前按完成情况打卡 | 普通用户未完成前序计划日 | 打开第 2 天计划，查看“按完成情况打卡”按钮 | 按钮禁用；绕过前端请求 `/checkin/today?planDay=2` 时后端返回前序打卡提示 |
| CHK-006 | 完成第 1 天后可打卡第 2 天 | 普通用户已完成第 1 天计划打卡 | 打开第 2 天，勾选动作并点击“按完成情况打卡” | 保存成功，`fitnesscheckins.plan_day` 记录为 2，复盘统计刷新 |
| CHK-007 | 同一天不能切换计划天数 | 今天已保存第 1 天计划打卡 | 直接请求保存第 2 天计划打卡 | 后端返回今天已记录第 1 天训练，不能改为第 2 天 |
| CHK-008 | 动作完成记录刷新后保留 | 普通用户已完成当前可打卡计划日中的部分动作 | 刷新页面后重新进入同一天训练计划 | 已完成动作仍保持勾选，完成比例不丢失 |
| CHK-009 | 训练复盘统计更新 | 普通用户保存一次训练打卡 | 进入“训练复盘” | 累计天数、平均时长、最近记录和热力图包含本次打卡 |
| CHK-010 | 连续打卡计算 | 准备连续日期的多条打卡记录 | 进入“训练复盘”或请求 `/checkin/stats` | 当前连续和最佳连续按日期连续段计算，历史记录按时间倒序返回 |
| CHK-011 | 计划洞察完成率计算 | 本周存在若干 `plantaskrecords` | 请求 `/fitnessplan/insight` | 返回本周记录动作数、完成动作数、完成率、恢复提示和下一步调整建议 |
| CHK-012 | 普通打卡不携带计划天数 | 普通用户已登录 | 在“训练打卡”页面直接保存今日打卡 | 保存成功，不触发计划日顺序限制；计划内打卡仍按 `planDay` 限制顺序 |
