## 一、课题背景与论文概述

### Q01 你的课题是做什么的？

- 简答：我的课题是设计并实现一个基于 Spring Boot 的健身规划软件，核心目标是把健身理论转成可计算规则，自动生成个性化训练计划，并配套动作指导、训练打卡、饮食建议和内容管理功能。
- 展开回答：它不是单纯记录运动数据的软件，而是先让用户填写健身目标、每周训练频率、可用器械和训练经验，再根据这些信息生成周期性训练计划。之后用户还能查看动作细节、完成打卡、看复盘统计、拿到饮食建议，所以它更像一个轻量的综合健身规划平台。
- 实现依据：论文摘要、`README.md` 的功能边界、前端 `mainpage.vue` 的用户端菜单、后端 `FitnessPlanController`、`NutritionController`、`FitnessCheckinController`、`ArticleController`。
- 继续追问时可补充：我把系统定位成“计划生成 + 执行辅助 + 训练记录”的闭环，而不是只做其中一个点。

### Q02 为什么选择这个题目？

- 简答：因为现有很多健身应用重记录、轻规划，能记录数据，但不一定能根据个人差异给出结构化训练计划。
- 展开回答：论文里提到，公众健身需求正在增长，但健身误区也很多。很多人知道自己想减脂或增肌，却不知道一周练几天、每次练什么、动作怎么选。我做这个题目，是想把健身目标、频率、器械和经验这些因素统一起来，让系统先生成一个可执行的基础计划。
- 实现依据：论文绪论、需求分析章节、`README.md` 的项目定位。
- 继续追问时可补充：这个题目的价值不在算法多复杂，而在于把训练理论工程化落地。

### Q03 这个系统主要解决了什么问题？

- 简答：主要解决了普通用户不会系统安排训练计划的问题。
- 展开回答：更具体地说，它解决了四个问题。第一，不知道怎么按目标和频率分配训练日。第二，不知道在当前器械条件下该练什么动作。第三，不知道动作的组数、次数和休息怎么安排。第四，计划生成后缺乏动作学习、打卡记录和饮食辅助。我的系统把这几步串起来了。
- 实现依据：论文摘要、第四章模块划分、`docs/功能实现与测试用例.md`。
- 继续追问时可补充：系统没有试图替代专业教练，而是给用户一个结构化、可执行、可跟踪的起点。

### Q04 这个系统的核心创新点是什么？

- 简答：核心创新点是把健身规划过程拆成了多层规则映射，而不是直接给一个固定模板。
- 展开回答：论文里把它概括成四层映射：健身目标与每周训练频率决定训练分化模式；训练分化模式决定动作模式集合；动作模式与器械条件决定具体动作；动作模式与训练经验决定训练参数。这样生成出来的计划，比直接写死一套“周一胸周二背”的模板更有针对性。
- 实现依据：论文摘要、系统设计与实现章节，后端 `FitnessPlanController` 中依次调用 `FitnessPlanRuleService`、`DayTrainService`、`TrainingParamService`、`ActionDetailService`、`ExerciseGuideService`。
- 继续追问时可补充：创新点更偏规则工程化，不是机器学习创新。

- 展开回答：论文里写了整体架构、模块设计和实现思路，但真正能支撑答辩的，还是仓库里的前后端代码、数据库表和测试文档。答辩时我会以论文为主线，以项目代码作为证据。
- 实现依据：论文目录、仓库结构、`docs/功能实现与测试用例.md`。

### Q06 系统有哪些主要功能模块？

- 简答：主要有用户与权限模块、健身需求模块、训练计划模块、动作指导模块、训练打卡与复盘模块、饮食推荐模块、文章公告模块和管理端模块。
- 展开回答：普通用户主要使用训练计划、动作指导、饮食建议、训练打卡、训练复盘、公告和文章阅读。管理员主要负责文章、公告、动作库、饮食模板和用户管理。两类角色共用同一套后端，但前端展示入口不同，后端权限校验也不同。
- 实现依据：`README.md` 的功能边界，`mainpage.vue` 的 `userViewKeys` 和 `adminViewKeys`，各控制器接口。
- 继续追问时可补充：这样设计可以复用账户体系和基础数据，同时保证管理操作不会暴露给普通用户。

### Q07 你的研究目标是什么？

- 简答：研究目标是实现一个能根据用户差异生成健身计划，并能支持执行、记录和辅助决策的完整系统。
- 展开回答：不是只证明“我能写一个前后端项目”，而是验证健身理论能不能通过规则映射，变成真实可运行的 Web 软件功能。从结果看，系统已经能完成计划生成、动作展示、打卡统计和饮食建议这条主链路。
- 实现依据：论文绪论、摘要、系统实现章节。
- 继续追问时可补充：我把目标定义得比较清楚，就是先做出规则驱动的个性化规划，而不是追求一个大而全的平台。

## 二、技术选型与架构设计

### Q09 为什么后端选择 Spring Boot？

- 简答：因为 Spring Boot 适合快速构建分层清晰、接口型的 Java Web 项目。
- 展开回答：本项目需要做用户认证、REST 接口、业务分层、数据库访问和跨域配置。Spring Boot 在这些方面生态成熟，开发效率高，也便于和 MyBatis、JWT 结合。对于毕业设计来说，它能让我把重点放在业务实现，而不是放在复杂配置上。
- 实现依据：`fitnessGuidance/pom.xml`、`FitnessGuidanceApplication.java`、各 `@RestController`。
- 继续追问时可补充：我没有选更重的微服务方案，因为项目规模没有大到必须拆服务。

### Q10 为什么前端选择 Vue 3 和 Element Plus？

- 简答：因为 Vue 3 上手快、响应式开发效率高，Element Plus 适合快速搭建管理端和表单界面。
- 展开回答：这个系统页面比较多，既有普通用户端，也有管理端，还包含表单、列表、弹窗、标签页等典型后台交互。Vue 3 适合把不同功能状态放在一个响应式页面里管理，Element Plus 可以减少很多基础 UI 工作量。
- 实现依据：`vue-project/package.json`、`mainpage.vue` 的页面结构。
- 继续追问时可补充：我保留了前后端分离结构，后期如果要换 UI 设计，后端接口基本不用动。

### Q11 为什么采用前后端分离？

- 简答：因为前后端分离能把界面展示和业务逻辑解耦，便于开发、维护和后续扩展。
- 展开回答：前端负责页面渲染和交互，后端负责接口、规则和数据。两边通过 HTTP 和 JSON 通信。这样做以后，如果将来要改成移动端、小程序端或者重新设计网页风格，后端核心逻辑仍然可以复用。
- 实现依据：前端 `services/*.js` 调用后端接口，后端各控制器统一返回 `Result`。
- 继续追问时可补充：这也是论文里 MVC 和分层架构结合的具体落地方式。

### Q12 为什么使用 MyBatis？

- 简答：因为本项目的业务规则和数据表结构都比较明确，MyBatis 对 SQL 可控性更强。
- 展开回答：训练计划、饮食模板、动作库、打卡统计这些模块都带有较强的业务表结构特征，我希望清楚知道每次查了什么、改了什么，所以选了 MyBatis，而不是更强调自动映射的 ORM。这样在做条件查询、统计和表结构扩展时更直接。
- 实现依据：`mapper` 目录、实体类、`schema.sql`。
- 继续追问时可补充：对于毕业设计来说，可控和直观比“省几行代码”更重要。

### Q13 为什么使用 JWT？

- 简答：因为 JWT 适合前后端分离场景，可以做无状态认证。
- 展开回答：用户登录成功后，后端生成 token，前端后续请求把 token 放在请求头或参数里。后端解析 token 拿到用户名，再决定是否允许访问。这样不需要在服务端额外维护 session，会话管理更轻量，也更适合 REST 风格接口。
- 实现依据：`UserController.login()`、`Jwtutil.java`、多个控制器里 `parseToken` 的逻辑。
- 继续追问时可补充：本项目没有做特别复杂的权限中心，所以 JWT 已经够用。

### Q14 你论文里写了 openGauss，但项目 README 里本地默认又是 H2，这怎么解释？

- 简答：论文强调的是系统设计支持 openGauss，当前本地开发为了降低部署门槛，默认跑 H2 文件库。
- 展开回答：从工程角度看，这两个并不矛盾。openGauss 适合说明关系型数据库设计和正式部署方向，H2 则适合毕业设计演示和本地快速启动。仓库里保留了 `application-opengauss.properties` 和 `database/opengauss-init.sql`，说明系统具备切换能力；`README.md` 里默认 H2，是为了让项目更容易跑起来。
- 实现依据：`README.md`、`fitnessGuidance/src/main/resources/application.properties`、`application-opengauss.properties`、`database/opengauss-init.sql`。
- 继续追问时可补充：如果老师问“你最终答辩演示用哪个”，我会回答本地演示优先用 H2，部署说明支持 openGauss。

### Q15 你的系统架构是什么？

- 简答：整体是前后端分离架构，后端内部采用 MVC 和分层设计。
- 展开回答：浏览器里的 Vue 页面属于前端 UI 层；Spring Boot 控制器、服务类属于业务逻辑层；数据库表属于数据存储层。后端内部又分为 Controller、Service、Mapper、Pojo 四层。这样的结构便于把请求处理、业务规则和数据访问拆开。
- 实现依据：论文第四章、`controller`、`service/impl`、`mapper`、`pojo` 目录。
- 继续追问时可补充：答辩时要强调“前后端分离”是系统级结构，“MVC”是后端内部组织方式，两者不是冲突关系。

### Q16 你是怎么解决跨域问题的？

- 简答：通过 Spring MVC 的 CORS 配置解决。
- 展开回答：前端开发地址是 `http://127.0.0.1:5173` 或 `http://localhost:5173`，后端在 8080 端口，所以开发时存在跨域问题。我在 `WebConfig` 中配置了允许的源、请求方法、请求头和凭据，保证前端可以正常访问接口。
- 实现依据：`fitnessGuidance/src/main/java/com/xiaolanshu/fitnessGuidance/controller/WebConfig.java`。
- 继续追问时可补充：这属于开发和部署中的常见工程问题，必须处理，否则前端即使接口正确也调不通。

### Q17 为什么要统一用 Result 类封装返回值？

- 简答：因为统一返回格式更方便前端处理成功和失败状态。
- 展开回答：如果每个接口各自返回不同结构，前端会写很多特殊判断。统一用 `Result.success()` 和 `Result.error()` 后，前端只要按统一协议取数据或显示错误消息，接口风格更稳定，也更利于后续扩展。
- 实现依据：后端多个控制器统一返回 `Result<T>`。
- 继续追问时可补充：这也是接口设计规范的一部分。

### Q18 为什么没有上微服务、Redis、消息队列这些更复杂的技术？

- 简答：因为项目规模和目标不需要，使用它们反而会提高复杂度。
- 展开回答：本项目是毕业设计，重点在于业务逻辑完整、架构清晰、功能可演示。当前模块数量、并发规模和数据量都不大，单体 Spring Boot 已经足够。用更复杂的中间件不会提升核心业务价值，反而会分散实现重点。
- 实现依据：当前仓库结构和单体应用组织方式。
- 继续追问时可补充：这是典型的工程取舍问题，不是技术越多越好，而是合适最重要。

## 三、需求分析与系统设计

### Q19 系统有哪些用户角色？

- 简答：主要有普通用户和管理员两类。
- 展开回答：普通用户关注训练执行，包括个人资料、健身需求、计划查看、动作指导、打卡、复盘、饮食和文章阅读。管理员关注维护，包括文章公告发布、动作库维护、饮食模板维护、用户审核和用户管理。
- 实现依据：`README.md` 功能边界、`mainpage.vue` 中用户端和管理端视图集合、`UserController` 中身份字段 `identity`。
- 继续追问时可补充：系统没有单独建管理员表，而是在用户表中通过 `identity` 区分角色。

### Q20 系统的核心业务流程是什么？

- 简答：用户注册登录后，先完善个人资料和健身需求，系统再生成训练计划，用户根据计划查看动作、完成打卡，并结合饮食建议执行训练。
- 展开回答：如果从完整链路看，可以概括为“账户建立 -> 参数录入 -> 计划生成 -> 动作学习 -> 打卡记录 -> 复盘反馈 -> 饮食辅助”。管理员则负责维护内容和规则资源，保证用户端能拿到有效动作、饮食模板和文章公告。
- 实现依据：`mainpage.vue` 的普通用户视图流转、后端对应控制器。
- 继续追问时可补充：这个流程说明系统不是孤立功能堆砌，而是有连续用户旅程的。

### Q21 功能性需求有哪些？

- 简答：功能性需求包括账户管理、训练计划生成、动作指导、训练打卡、饮食推荐、文章公告和后台管理。
- 展开回答：如果细分，前台至少包括登录注册、资料维护、健身需求维护、计划查看、动作搜索与详情、打卡、复盘、饮食建议、公告和文章详情。后台则包括用户审核、文章管理、公告管理、动作库管理和饮食模板管理。
- 实现依据：论文需求分析章节、`README.md`、`docs/验收清单.md`。
- 继续追问时可补充：这些需求最后都能对应到仓库里的具体页面和接口。

### Q22 非功能性需求有哪些？

- 简答：主要包括可维护性、可用性、安全性和基本性能要求。
- 展开回答：可维护性体现在前后端分离和分层结构；可用性体现在界面统一、普通用户和管理员入口清晰；安全性体现在 JWT 校验和管理员权限检查；性能方面，本项目数据规模不大，主要保证常见页面和查询在本地环境下稳定响应。
- 实现依据：论文需求分析、系统设计章节、权限校验代码。
- 继续追问时可补充：毕业设计通常不会做高并发压测，但要把设计上的基本考虑说清楚。

### Q23 为什么要把管理员审核设计进去？

- 简答：因为系统默认不希望所有注册账户都直接生效，审核能增加管理控制。
- 展开回答：在 `UserController` 里，普通用户注册后并不是天然可以使用全部功能，管理员可以通过审核控制注册状态。通过后系统还会自动为新用户创建默认健身需求；撤回时会清理对应资料。这体现了管理员对用户生命周期的控制。
- 实现依据：`UserController.editregister()`。
- 继续追问时可补充：这个设计也让管理端不是摆设，而是确实参与业务流程。

### Q24 为什么动作库、饮食模板也要做成可维护的后台资源？

- 简答：因为这些内容是计划和建议的基础资源，写死在前端或代码里不利于扩展。
- 展开回答：动作指导、食材、餐单模板、替换规则和饮食场景，本质上都属于知识库资源。把它们放到数据库并交给管理员维护，后期新增动作或调整饮食建议时，不需要直接改前端页面文本。
- 实现依据：`ExerciseGuideController`、`NutritionController` 的管理员接口，`schema.sql` 中 `exerciseguides`、`fooditems`、`mealtemplates`、`foodreplacements`、`eatingscenarios`。
- 继续追问时可补充：这也是系统从“演示型”往“产品化”迈出的一步。

### Q25 为什么训练计划要支持提前查看后续天数，但不允许提前打卡？

- 简答：因为查看有助于用户预习训练安排，但提前打卡会破坏执行顺序和统计准确性。
- 展开回答：从产品体验看，用户可能想先看后几天练什么，所以允许查看是合理的；但从数据一致性看，如果前面的计划日没完成，却先打了后面的卡，会让训练周期和复盘统计失真，所以系统只开放查看，不开放越序打卡。
- 实现依据：`mainpage.vue` 的锁定逻辑，`FitnessCheckinServiceImpl.validatePlanDayOrder()`。
- 继续追问时可补充：这是一个“可用性”和“数据约束”之间的平衡设计。

### Q26 为什么文章和公告也放在健身规划系统里？

- 简答：因为训练计划之外，用户还需要持续获取内容和通知，文章和公告可以承担辅助教育和信息发布作用。
- 展开回答：训练类软件如果只有计划，没有解释和内容，会比较干。文章可以用来补充训练知识、饮食知识、动作原理；公告则适合做通知和提醒。这样系统从单一工具变成了更完整的健身服务平台。
- 实现依据：论文摘要中提到文章与公告扩展功能，`ArticleController`、`NoticeController`、前端文章广场和公告中心页面。
- 继续追问时可补充：这部分不是核心算法，但能增强系统完整性。

## 四、核心实现题

### Q27 用户注册和登录是怎么实现的？

- 简答：用户注册通过 `/user/register` 写入账户，登录通过 `/user/login` 校验密码，成功后返回 JWT。
- 展开回答：注册时会先检查用户名和密码是否为空、是否超长、用户名是否已占用。登录时会根据用户名查用户，使用 `PasswordUtil` 校验密码，通过后再检查 `registered` 是否已审核。审核通过才生成 token，后续接口通过 token 识别当前用户。
- 实现依据：`UserController.register()`、`UserController.login()`、`PasswordUtil.java`、`Jwtutil.java`、`users` 表。
- 继续追问时可补充：密码不是明文比较，系统兼容了旧密码数据，并在合适时机升级为哈希存储。

### Q28 个人资料功能是怎么实现的？

- 简答：前端个人资料页填写昵称、密码、身高、体重等信息，调用 `/user/editmessage` 更新当前登录用户数据。
- 展开回答：个人资料是账户基础数据，和健身需求不同。它主要改的是用户实体里的昵称、身高、体重、擅长方向等字段。后端先根据 token 确认当前用户身份，再只允许修改本人信息，不允许借前端传入的用户名改别人数据。
- 实现依据：`mainpage.vue` 的 `profileForm`，`UserController.editmessage()`，`users` 表。
- 继续追问时可补充：这里体现了“以 token 用户为准，而不是以前端传值为准”的安全思想。

### Q29 健身需求功能是怎么实现的？

- 简答：用户在前端填写健身目标、每周频率、器械和训练经验，后端保存到用户健身偏好表，后续训练计划和饮食推荐都从这里取值。
- 展开回答：这部分数据不是账户身份数据，而是业务输入数据。系统把它单独抽成 `userprofiles` 表，原因是它会频繁参与训练计划生成和饮食推荐计算，而且和账户信息不是一类概念。
- 实现依据：`UserProfileController`、`UserProfileService`、`userprofiles` 表、`mainpage.vue` 的健身需求页面。
- 继续追问时可补充：把它拆出来也方便管理员后期做用户画像和扩展分析。

### Q30 个性化训练计划是怎么生成的？

- 简答：系统先根据用户目标和频率确定训练分化模式，再根据每天的动作模式、器械条件和训练经验，组合出当天动作和参数。
- 展开回答：前端进入训练计划页后，会调用 `/fitnessplan/getsplitmode` 和 `/fitnessplan/getactiontask`。后端先读取当前用户 `userprofiles`，再从 `fitnessplanrules` 找到对应分化模式，从 `daytrains` 找到当天动作模式集合，从 `trainingparams` 取组数次数休息区间，从 `actiondetails` 和 `exerciseguides` 取动作说明和具体动作。最后把这些拼成 `ActionTask` 列表返回前端。
- 实现依据：`trainingApi.js`、`FitnessPlanController.getsplitMode()`、`FitnessPlanController.getactionTask()`、`userprofiles`、`fitnessplanrules`、`daytrains`、`trainingparams`、`actiondetails`、`exerciseguides`。
- 继续追问时可补充：这就是论文里“四层映射”的代码化实现。

### Q31 训练分化模式是什么，为什么要先确定它？

- 简答：训练分化模式可以理解为一周训练日怎么分配，比如不同天练不同肌群或动作重点。
- 展开回答：如果不先确定分化模式，系统就不知道一周训练结构怎么组织。先根据目标和频率定结构，再为每天填动作和参数，逻辑会更稳定。这样计划不是零散动作堆起来的，而是有周节奏的。
- 实现依据：`FitnessPlanController.getsplitMode()`，`fitnessplanrules` 表和相关服务。
- 继续追问时可补充：这是计划生成的第一层，决定的是“框架”，不是具体动作。

### Q32 为什么计划生成不直接写死具体动作？

- 简答：因为同一个动作模式在不同器械条件下可以有不同具体动作，直接写死会降低适配性。
- 展开回答：比如都属于推类动作，徒手、哑铃、杠铃、器械条件下可选动作不同。系统先确定动作模式，再根据器械去匹配动作指导，更能适配家庭、宿舍、健身房等场景。
- 实现依据：`ActionDetailService`、`ExerciseGuideService`、`exerciseguides` 表的 `actionPattern` 和 `equipment` 维度。
- 继续追问时可补充：这也是为什么论文里把“动作模式”和“具体动作”分开设计。

### Q33 训练参数是怎么给出的？

- 简答：训练参数来自动作模式和训练经验的映射。
- 展开回答：后端根据动作模式和 `exLevel` 查询 `trainingparams`，返回组数、次数和休息时间范围。这样新手和进阶用户即使练同一类动作，也能拿到不同强度建议，不会完全一样。
- 实现依据：`FitnessPlanController.getactionTask()` 中调用 `trainingParamService.gettrainingparam()`，`trainingparams` 表。
- 继续追问时可补充：这个参数不是实时 AI 算出来的，而是规则库映射出来的，更可控。

### Q34 动作指导功能是怎么实现的？

- 简答：动作指导通过动作模式和器械查询具体动作记录，返回动作名称、步骤、要点、图片和风险提示。
- 展开回答：普通用户可以在动作指导页搜索动作模式和器材，后端 `/exerciseguide/getexerciseguide` 返回一条 `ExerciseGuide`。除了基本描述，系统还支持难度、主肌群、辅助肌群、常见错误、禁忌、适合人群和替代动作这些字段，信息比简单图文更完整。
- 实现依据：`ExerciseGuideController.getexerciseguide()`、`exerciseguides` 表、前端动作指导页面。
- 继续追问时可补充：这部分的价值在于把“计划里出现的动作”解释清楚，降低用户执行门槛。

### Q35 动作替换功能为什么要做？

- 简答：因为同一个动作模式下，用户可能临时没有器械、动作不适应，或者想换一个等价动作。
- 展开回答：系统支持在训练计划动作卡片中做替换。前端调用 `/fitnessplan/action/replace` 或动作替代列表接口，后端按照动作模式、偏好器材和当前动作 ID 查找替代动作。这样不会改变训练目的，只改变具体执行动作。
- 实现依据：`FitnessPlanController.replaceAction()`、`ExerciseGuideController.alternatives()`、`exercisealternatives` 表与 `ExerciseGuideService`。
- 继续追问时可补充：这说明系统不是死板输出，而是保留了一定灵活性。

### Q36 训练打卡是怎么实现的？

- 简答：训练打卡通过 `/checkin/today` 保存用户当天的训练时长、状态、备注和计划天数。
- 展开回答：系统支持两种打卡方式，一种是普通今日打卡，一种是从训练计划页按完成情况自动生成打卡。后端会先检查今天是否已有记录，有则更新，没有则插入。这样同一天只保留一条有效打卡，避免重复堆积。
- 实现依据：`trainingApi.js` 的 `saveTodayCheckin()`，`FitnessCheckinController`，`FitnessCheckinServiceImpl.saveToday()`，`fitnesscheckins` 表。
- 继续追问时可补充：计划内打卡会额外记录 `planDay`，方便顺序校验和后续统计。

### Q37 为什么打卡要限制训练计划顺序？

- 简答：因为训练计划本身有周期顺序，如果跳着打卡，会导致统计和调整建议失真。
- 展开回答：后端 `validatePlanDayOrder()` 会检查当前要打的 `planDay` 之前是否都已经打过。比如第 1 天没完成，就不能直接打第 2 天。这样设计是为了让打卡数据更真实，也让训练周期、完成率和计划建议更有意义。
- 实现依据：`FitnessCheckinServiceImpl.validatePlanDayOrder()`，`fitnesscheckins.plan_day` 字段。
- 继续追问时可补充：前端也做了按钮禁用，但真正的约束放在后端，避免被绕过。

### Q38 训练计划中的动作完成状态是怎么保存的？

- 简答：动作完成状态保存在 `plantaskrecords` 表中。
- 展开回答：用户在训练计划页勾选某个动作完成后，前端调用 `/fitnessplan/taskrecord` 保存记录，内容包括用户名、计划日期、训练日、动作索引和完成状态。刷新页面后，前端再用 `/fitnessplan/taskrecords` 把已保存的状态读回来，所以勾选不会丢失。
- 实现依据：`trainingApi.js` 的 `savePlanTaskRecord()`、`getPlanTaskRecords()`，`FitnessPlanController.taskRecords()`、`saveTaskRecord()`，`plantaskrecords` 表。
- 继续追问时可补充：这个表保存的是动作级完成情况，不是整天是否打卡，两者粒度不同。

### Q39 训练复盘功能是怎么实现的？

- 简答：训练复盘主要根据打卡记录统计累计天数、时长、连续天数、最佳连续和最近记录。
- 展开回答：后端 `/checkin/stats` 会统计总打卡天数、累计分钟数、本月天数、平均时长，并取最近记录。`FitnessCheckinServiceImpl` 里还写了当前连续和最佳连续的计算逻辑。前端再把这些数据做成指标卡片、热力图和建议展示。
- 实现依据：`FitnessCheckinServiceImpl.getStats()`，`mainpage.vue` 的复盘页，`fitnesscheckins` 表。
- 继续追问时可补充：当前连续和最佳连续不是数据库直接存的，而是服务层根据日期列表计算的。

### Q40 饮食推荐功能是怎么实现的？

- 简答：后端根据体重、训练目标和每周训练频率计算目标热量、蛋白质、脂肪、碳水和饮水量，再拼装餐次和建议。
- 展开回答：前端饮食建议页加载时会调用 `/nutrition/recommendation` 和 `/nutrition/library`。后端先从 `users`、`userprofiles` 和兼容保留的 `nutritionpreferences` 读取数据，然后在 `buildRecommendation()` 里按规则计算热量和宏量营养素，再生成训练日提醒、休息日提醒、餐次建议、食物选择、替换建议和采购清单。
- 实现依据：`nutritionApi.js`，`NutritionController.recommendation()`，`NutritionController.buildRecommendation()`，`users`、`userprofiles`、`nutritionpreferences`。
- 继续追问时可补充：它是规则型推荐，不是医学营养系统，所以定位是训练辅助参考。

### Q41 饮食推荐里的模板库是怎么来的？

- 简答：来自管理员维护的食材、餐单模板、替换规则和饮食场景数据。
- 展开回答：系统把饮食辅助资源拆成四类表：`fooditems`、`mealtemplates`、`foodreplacements`、`eatingscenarios`。普通用户端读取 `/nutrition/library` 时，后端会结合用户目标和偏好做筛选；管理员端用 `/nutrition/admin/library` 查看完整库并做维护。
- 实现依据：`NutritionController.library()`、`adminLibrary()`、`NutritionLibraryMapper`、相关数据表。
- 继续追问时可补充：这样系统既有计算型建议，也有内容型建议。

### Q42 文章功能是怎么实现的？

- 简答：文章功能由管理员发布，普通用户可搜索和查看详情。
- 展开回答：管理员在前端内容管理页填写标题、主题和正文，调用 `/article/addarticle` 发布。普通用户在文章广场通过标题和主题搜索，调用 `/article/getarticle` 获取列表，点击文章后再通过 `/article/getarticledetail` 查看详情。写接口都要求管理员身份。
- 实现依据：`contentApi.js`、`mainpage.vue` 的文章页面、`ArticleController`、`articles` 表。
- 继续追问时可补充：这个模块体现了内容管理和普通阅读场景的分离。

### Q43 公告功能是怎么实现的？

- 简答：公告和文章类似，也是管理员发布、普通用户查看。
- 展开回答：区别在于公告更偏通知型内容，支持草稿保存、载入和清空，方便管理员反复编辑。普通用户只能在公告中心查看已发布结果，不会看到编辑入口。
- 实现依据：`NoticeController`、`contentApi.js` 的公告接口、`notices` 和 `tempnotices` 表。
- 继续追问时可补充：临时草稿表说明我考虑了管理端实际使用场景，而不是只做最简单的发布。

### Q44 管理员权限是怎么控制的？

- 简答：后端根据 token 解析出当前用户名，再查询用户身份是否为 `ADMIN`。
- 展开回答：像文章新增、公告发布、动作维护、饮食模板维护、用户管理这些接口，都不会只信前端页面是否隐藏按钮，而是后端再次查 `users.identity`。如果不是管理员，就统一返回“无相关权限”。
- 实现依据：`UserController`、`ArticleController`、`ExerciseGuideController`、`NutritionController` 中的管理员校验逻辑。
- 继续追问时可补充：这说明权限控制的真正边界在后端，不在前端。

### Q45 管理端用户管理是怎么实现的？

- 简答：管理员可以查看所有用户、添加用户、审核注册、编辑用户信息和删除用户。
- 展开回答：`/user/getallmessage` 返回用户列表；`/user/adduser` 新增用户并创建默认健身需求；`/user/editregister` 控制审核状态，通过时创建默认资料，撤回时清理资料；`/user/deleteuser` 删除用户时还会同步删除用户资料和文章，保持数据整洁。
- 实现依据：`UserController.getallmessage()`、`adduser()`、`editregister()`、`deleteuser()`。
- 继续追问时可补充：这里能体现后台管理对业务数据完整性的影响。

## 五、数据库、接口与工程化

### Q46 为什么要把用户表和用户健身偏好表分开？

- 简答：因为账户信息和健身业务参数不是同一类数据。
- 展开回答：用户表主要负责登录、身份、基础资料；健身偏好表负责训练目标、频率、器械和经验。拆开后结构更清晰，也避免把业务变化频繁的字段全部塞进用户主表。
- 实现依据：`users` 表、`userprofiles` 表。
- 继续追问时可补充：这也是常见的“账户域”和“业务画像域”分离思想。

### Q47 训练计划相关表为什么要拆这么多？

- 简答：因为训练计划本身有多层语义，拆表后更容易表达规则和复用。
- 展开回答：`fitnessplanrules` 管的是目标和频率对应什么分化模式；`daytrains` 管的是某个分化模式包含哪些动作模式；`trainingparams` 管的是动作模式和经验对应什么参数；`actiondetails` 和 `exerciseguides` 管的是描述和具体动作内容。拆表以后，规则可以灵活组合，不是一张大表全写死。
- 实现依据：`schema.sql` 中相关表结构，`FitnessPlanController` 的查询链路。
- 继续追问时可补充：这就是“规则资源化”的数据库体现。

### Q48 为什么动作完成记录和打卡记录不能合成一张表？

- 简答：因为它们表示的是两个不同粒度的业务对象。
- 展开回答：动作完成记录关注某天某个动作有没有完成，粒度更细；打卡记录关注某天整体训练有没有完成、状态怎样、练了多久，粒度更粗。如果强行合并，会让统计逻辑和数据含义都变混乱。
- 实现依据：`plantaskrecords` 表、`fitnesscheckins` 表。
- 继续追问时可补充：动作级数据适合分析执行细节，日级数据适合分析习惯和趋势。

### Q49 后端接口设计有什么特点？

- 简答：接口按业务模块划分，基本采用 REST 风格，并统一返回 `Result`。
- 展开回答：比如用户相关都放在 `/user`，训练计划相关在 `/fitnessplan`，打卡在 `/checkin`，饮食在 `/nutrition`，文章在 `/article`。这样接口边界清楚，前端也更容易按模块封装 `services`。
- 实现依据：各控制器的 `@RequestMapping`，前端 `services/*.js`。
- 继续追问时可补充：虽然不是完全严格的 REST 资源建模，但对毕业设计规模来说已经足够清晰。

### Q50 前端和后端是怎么交互的？

- 简答：前端通过 Axios 请求后端接口，后端返回 JSON，前端再把数据渲染到页面。
- 展开回答：前端在 `services` 目录中把接口调用封装成函数，比如训练模块、饮食模块、内容模块各自独立。页面逻辑层只调用这些函数，不直接写请求细节。后端收到请求后解析参数、校验 token、运行服务逻辑，再返回统一格式。
- 实现依据：`trainingApi.js`、`nutritionApi.js`、`contentApi.js`、各控制器。
- 继续追问时可补充：这样做还有一个好处，就是接口改动时只需集中改服务层封装。

### Q51 图片上传和展示是怎么处理的？

- 简答：管理员上传动作图片后，后端把文件保存到本地上传目录，再通过静态资源映射对外访问。
- 展开回答：`/exerciseguide/uploadimage` 先校验是不是图片类型，再生成唯一文件名，保存到 `uploads/exercise-guides`。`WebConfig` 中又把 `/uploads/**` 映射到本地上传目录，所以前端拿到图片路径后能直接显示。
- 实现依据：`ExerciseGuideController.uploadImage()`、`WebConfig.addResourceHandlers()`。
- 继续追问时可补充：这样静态资源和数据库记录分开，数据库只存路径，不存二进制文件。

### Q52 为什么本地上传目录不直接提交到仓库？

- 简答：因为上传文件属于运行时资源，不适合和源码一起长期维护。
- 展开回答：仓库里只保留了 `.gitkeep` 占位，真正上传的图片在运行时生成。如果把这些文件全提交，会让仓库越来越重，也会让演示数据和源码耦合过紧。
- 实现依据：`README.md`、`fitnessGuidance/uploads/exercise-guides/.gitkeep`。
- 继续追问时可补充：这是常见的软件工程规范，不是这个项目特有问题。

### Q53 你是怎么保证接口安全性的？

- 简答：主要通过登录 token 校验和管理员权限二次校验。
- 展开回答：大多数需要登录的接口都会先解析 token；涉及写操作和后台资源的接口，还会进一步校验当前用户是否为管理员。另外，个人资料修改等接口不会信任前端传的用户名，而是以 token 中的当前用户为准。
- 实现依据：`Jwtutil`、多个控制器的 token 和管理员校验逻辑。
- 继续追问时可补充：当前项目的安全设计属于课程和毕业设计可接受范围，但还没有做到企业级细粒度授权。

### Q54 如果 token 失效或者用户未登录，会发生什么？

- 简答：后端会返回“未登录”，前端据此提示用户重新登录。
- 展开回答：多个控制器都把解析 token 作为入口校验，如果解析失败就直接返回错误，不继续执行业务逻辑。这样可以避免未授权请求进入后续处理。
- 实现依据：`UserController` 以外多个控制器里的 `parseToken` 或 `parseUsername` 逻辑。
- 继续追问时可补充：这也是无状态认证的典型处理方式。

### Q55 为什么要写单独的测试文档和验收清单？

- 简答：因为代码能跑不等于系统可验证，测试文档能把功能路径和预期结果固定下来。
- 展开回答：`docs/功能实现与测试用例.md` 更细，适合说明每个功能怎么实现、怎么测；`docs/验收清单.md` 更偏回归验证，适合每次改完快速检查核心功能有没有回退。这说明项目不仅停留在“写出来”，也考虑了“怎么验证”。
- 实现依据：两个文档本身。
- 继续追问时可补充：在答辩里提这两个文档，会显得工程意识更完整。

### Q56 你如何证明训练计划、饮食推荐这些功能是可测的？

- 简答：因为它们都有明确输入和可预期输出，可以设计规则型测试用例。
- 展开回答：比如训练计划可以测不同目标、频率、器械和经验组合下返回的分化模式、动作列表和参数范围是否合理；饮食推荐可以测目标热量是否下限受控、不同目标是否有不同宏量营养分配；打卡可以测顺序限制、重复更新和统计结果。
- 实现依据：`docs/功能实现与测试用例.md` 中的用例设计，相关控制器与服务逻辑。
- 继续追问时可补充：这类系统不是黑盒 AI，所以更适合做规则验证。

### Q57 你的系统目前有哪些工程化特点？

- 简答：主要有前后端分离、统一接口返回、角色权限区分、资源后台化、测试文档化和本地快速启动脚本。
- 展开回答：这些特点说明项目不是只为了把页面做出来，而是有一定可维护性。比如 `start-dev.command` 方便启动，`services` 目录统一接口调用，管理员资源维护让系统更可扩展，测试文档让验证更可复用。
- 实现依据：仓库结构、`README.md`、前后端代码组织。
- 继续追问时可补充：如果老师问“你项目哪里体现软件工程”，这题很适合回答。

### Q58 你觉得这个项目最有代表性的工程取舍是什么？

- 简答：最有代表性的取舍是使用规则驱动和资源表配置，而不是上复杂算法。
- 展开回答：对于健身规划这种强业务约束场景，先把规则和资源建好，比一开始追求复杂推荐模型更稳。规则型系统的优点是可解释、可调试、可验证，也更适合毕业设计展示实现过程。
- 实现依据：训练计划生成链路、饮食推荐规则、资源库设计。
- 继续追问时可补充：这不是说算法不重要，而是现阶段先把基础工程链路做扎实。

## 六、测试、难点、不足与展望

### Q59 你的系统做了哪些测试？

- 简答：主要做了功能测试、权限测试、异常测试和基础回归测试。
- 展开回答：功能测试覆盖登录注册、训练计划、动作指导、打卡、饮食、文章公告和后台管理；权限测试主要验证普通用户不能调用后台写接口；异常测试包括未登录、重复用户名、缺少必填参数、上传非法图片、访问不存在文章等；回归测试则依靠验收清单快速检查主流程。
- 实现依据：`docs/功能实现与测试用例.md`、`docs/验收清单.md`。
- 继续追问时可补充：当前测试以手工和接口逻辑验证为主，自动化测试还不够完整。

### Q60 项目实现过程中的难点是什么？

- 简答：主要难点是把抽象训练理论拆成可计算规则，并让前后端展示出来。
- 展开回答：难点不只是写接口，而是怎么把“目标、频率、器械、经验”这些概念分层组织成训练计划。如果分层不清，数据库会混乱，代码也很难维护。另一个难点是让计划不只是动作名，还能带出动作细节、参数和替代方案，形成完整体验。
- 实现依据：训练计划相关表设计、控制器组合逻辑、动作库扩展字段。
- 继续追问时可补充：这题最好强调“业务建模难于写页面”。

### Q61 你的系统有哪些不足？

- 简答：主要不足是智能化程度还有限，推荐规则还比较静态，测试自动化也不够。
- 展开回答：第一，训练计划和饮食推荐主要基于规则映射，还没有结合更多历史行为做动态学习。第二，饮食推荐属于一般性建议，不能替代专业营养方案。第三，自动化测试覆盖不够，更多还是手工验证。第四，部署形态偏本地演示，还没有做真正线上化运维设计。
- 实现依据：当前代码和文档边界。
- 继续追问时可补充：答辩时主动承认不足，比硬说“已经很完善”更稳。

### Q62 如果继续迭代，你会先做什么？

- 简答：我会先增强动态调整能力和自动化测试。
- 展开回答：动态调整方面，可以根据最近打卡、完成率、状态反馈去微调后续计划强度，而不只是给出提示。测试方面，可以补前后端自动化用例，减少每次大改后的人工回归成本。之后再考虑更细的用户画像和更丰富的推荐逻辑。
- 实现依据：当前已有 `insight`、`adjustments`、打卡统计基础。
- 继续追问时可补充：这说明当前项目已经为后续迭代留了接口和数据基础。

### Q63 这个项目为什么说是“健身理论工程化”？

- 简答：因为它把原本只能由人理解的训练原则，拆成了程序能执行的规则和表结构。
- 展开回答：传统训练理论通常以书籍、经验和文字建议存在，不容易直接变成软件行为。我的项目把这些原则转成了目标、频率、分化模式、动作模式、参数和动作资源之间的映射，最后由系统自动算出用户能看的训练计划。
- 实现依据：论文摘要、训练计划表设计、`FitnessPlanController` 逻辑。
- 继续追问时可补充：这是本课题最核心的学术表达，也是最适合强调的价值点。

### Q64 你的项目更偏“管理系统”还是“智能推荐系统”？

- 简答：我认为它本质上是一个规则驱动的健身规划系统，兼有部分管理系统特征。
- 展开回答：它不像传统后台管理系统那样只是增删改查，也不像纯推荐系统那样完全靠模型学习。它的核心在于规则驱动的个性化规划，管理端只是支撑资源维护和角色控制。
- 实现依据：功能边界、计划生成逻辑、后台维护模块。
- 继续追问时可补充：这个回答可以避免老师把你项目误解成“普通信息管理系统”。

### Q65 为什么你的项目不算简单的课程作业拼凑？

- 简答：因为它有完整业务闭环、清晰数据建模和多模块联动，不是几个页面和 CRUD 的简单组合。
- 展开回答：如果只是课程作业，往往做到登录、列表、增删改查就结束了。我的项目不仅有这些基础能力，还把训练计划生成、动作指导、打卡统计、饮食建议和后台资源维护串成了连续业务链，而且每一块都有对应数据表和接口支撑。
- 实现依据：论文摘要、仓库结构、实现与测试文档。
- 继续追问时可补充：答辩时这题可以结合“规则工程化”一起讲。

### Q66 如果老师质疑“这只是规则匹配，不够智能”，你怎么回答？

- 简答：我会承认当前是规则驱动系统，但这恰恰是本项目阶段性的合理选择。
- 展开回答：第一，规则驱动并不代表没有价值，很多专业系统前期都先从规则化开始。第二，健身规划本身就有较强专业约束，可解释性很重要。第三，毕业设计更强调系统分析、设计、实现和验证，规则驱动更便于证明系统逻辑是清楚的。未来完全可以在此基础上引入更动态的推荐策略。
- 实现依据：当前训练计划和饮食推荐实现方式。
- 继续追问时可补充：这个回答要避免和老师硬碰硬，重点是“现阶段选择合理”。

### Q67 如果老师问“你的系统和 Keep 一类产品比，有什么不同”，怎么答？

- 简答：它们更偏内容和课程产品，我的系统更强调基于个人输入生成结构化训练计划。
- 展开回答：现有成熟产品往往内容丰富、生态强，但很多功能是课程推荐、打卡社区或视频跟练。我的系统重点不在海量内容，而在于把目标、频率、器械和经验变成可执行的周期计划，并用动作库、打卡和饮食建议辅助执行。
- 实现依据：项目核心链路设计。
- 继续追问时可补充：可以说我的项目更像“规划型原型系统”，而不是商业化成熟产品的全量复制。

### Q68 如果老师问“你最大的收获是什么”，怎么答？

- 简答：最大的收获是学会了把抽象业务问题拆成可落地的数据结构、接口和页面流程。
- 展开回答：这次不是单纯学了某个框架，而是完整经历了一遍需求分析、表设计、接口设计、前后端联调、测试验证和文档总结。尤其是在训练计划这块，我真正体会到业务建模的重要性。

## 七、专项题：功能是怎么实现的

### 1. 训练计划功能怎么实现

- 口语模板：前端进入训练计划页后，会先请求当天训练分化模式和动作任务列表。后端先根据当前登录用户去查健身需求，再用健身目标和每周频率匹配训练分化模式；然后根据当天属于什么分化模式，去查这一天应该有哪些动作模式；再根据动作模式和训练经验匹配组数、次数、休息时间，根据动作模式和器械条件匹配具体动作、步骤、要点和图片，最后把这些内容拼成前端展示的动作卡片。
- 对应接口：`/fitnessplan/getsplitmode`、`/fitnessplan/getactiontask`
- 对应表：`userprofiles`、`fitnessplanrules`、`daytrains`、`trainingparams`、`actiondetails`、`exerciseguides`

### 2. 训练打卡功能怎么实现

- 口语模板：训练打卡有普通打卡和计划内打卡两种入口。普通打卡是用户手动填时长、状态和备注；计划内打卡是用户在训练计划里勾选完成动作后，系统自动带出时长和完成比例再提交。后端收到请求后会先检查今天有没有记录，有就更新，没有就新增；如果这次打卡带了计划天数，还会检查前面的计划日是不是已经完成，防止用户跳着打卡。
- 对应接口：`/checkin/today`、`/fitnessplan/taskrecord`、`/fitnessplan/taskrecords`
- 对应表：`fitnesscheckins`、`plantaskrecords`

### 3. 饮食推荐功能怎么实现

- 口语模板：前端打开饮食建议页后，会分别请求推荐结果和饮食资源库。后端先读取当前用户的体重、健身目标、每周训练频率和兼容保留的偏好数据，再按规则计算目标热量、蛋白质、脂肪、碳水和饮水量，然后拼出训练日提醒、休息日提醒、餐次建议、食材选择、替换规则和采购清单。除此以外，管理员维护的餐单模板、食材和外食场景也会参与展示。
- 对应接口：`/nutrition/recommendation`、`/nutrition/library`
- 对应表：`users`、`userprofiles`、`nutritionpreferences`、`fooditems`、`mealtemplates`、`foodreplacements`、`eatingscenarios`

### 4. 文章管理功能怎么实现

- 口语模板：管理员在内容管理页填写标题、主题和正文后，前端调用新增文章接口。后端先根据 token 确认当前登录人是不是管理员，校验通过后再把文章写入文章表。普通用户只能调用文章列表和详情接口，通过标题和主题检索文章，不能调用新增、编辑和删除接口。
- 对应接口：`/article/addarticle`、`/article/getarticle`、`/article/getarticledetail`、`/article/editarticle`、`/article/deletearticle`
- 对应表：`articles`

### 5. 管理员权限控制怎么实现

- 口语模板：系统不是只靠前端隐藏按钮来做权限控制，而是每次管理接口请求进来后，后端都会解析 token，查当前用户名，再查用户表中的身份字段。如果身份不是 `ADMIN`，接口直接返回“无相关权限”。所以就算普通用户手动构造请求，也不能真正写入管理数据。
- 对应接口：用户管理、文章管理、公告管理、动作库管理、饮食模板管理相关写接口
- 对应表：`users`

## 八、答辩时容易说错的点

- 不要把“前后端分离架构”和“后端 MVC 架构”说成同一个概念。
- 不要把“openGauss 支持”和“本地默认 H2”说成互相矛盾，正确说法是开发默认 H2，设计与部署支持 openGauss。
- 不要把训练计划说成“AI 自动生成”，更准确的说法是“基于规则映射自动生成”。
- 不要把饮食推荐说成专业医疗营养方案，正确说法是训练辅助型饮食建议。
- 不要把管理员权限说成前端控制，真正的权限边界在后端。
- 不要忽略测试文档，否则老师会觉得项目验证不充分。
- 不要只讲页面，不讲接口和数据表；也不要只讲表，不讲用户使用流程。

## 九、最后建议

- 如果老师问得很宽泛，先按“做什么 -> 为什么做 -> 怎么实现 -> 效果怎样 -> 还有什么不足”五步回答。
- 如果老师问的是功能实现，尽量套用“前端入口 -> 后端接口 -> 业务逻辑 -> 数据表 -> 异常处理”这条链。
- 如果老师质疑智能性，不要硬辩成 AI，直接承认它是规则驱动，但强调可解释、可验证、适合毕业设计阶段。
- 如果老师追到数据库，优先讲为什么拆表，而不是先背字段名。
- 如果老师追到项目价值，优先强调“健身理论工程化”和“从计划到执行到记录的闭环”。
