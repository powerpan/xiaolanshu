# 小蓝鼠产品化升级计划

本文档用于记录小蓝鼠指导系统的后续产品化方向。范围聚焦产品功能、前端体验和当前代码结构整理，不包含 Docker、CI、生产部署、线上运维等工程化事项。

## 当前执行状态

- 阶段一已完成：README 已同步当前账号、功能边界、动作图片上传方式和验证命令；已新增 [功能验收清单](验收清单.md)；旧版未路由页面和 Vite 示例组件已清理。
- 阶段二已完成首轮重构：前端已拆出侧边栏、顶栏、用户端/管理端页面壳、指标卡、热力图组件；接口调用已按用户、内容、训练、动作库、饮食拆到 `services` 目录，后端接口名称保持不变。
- 阶段三已完成本轮产品化深化：动作库已增加内容完整度评分、缺替代动作筛选和管理员上传图片维护；训练计划已增加训练周期、恢复日、计划调整记录和同模式动作替换；饮食推荐已接入食材、餐单模板、替换规则和外食场景数据，并新增管理员饮食模板维护入口。
- 品牌化和交付文档已补齐：系统名称改为小蓝鼠，前端接入纯色品牌标识、暖色页面背景和背景音乐光碟；README 增加 Windows 交付入口，另新增 [Windows 部署操作流程](Windows部署操作流程.md) 和 [功能实现与测试用例](功能实现与测试用例.md)。
- 下一阶段建议：继续把 `mainpage.vue` 拆到训练、动作、饮食、内容和管理模块中，避免后续新逻辑继续堆回单个页面文件。

## 当前项目状态

当前项目已经从基础 CRUD 演示扩展为一个前后端分离的健身指导产品雏形：

- 普通用户端：登录注册、首页、个人资料、健身需求、训练计划、动作指导、训练打卡、训练复盘、饮食建议、公告中心、文章广场和详情阅读。
- 管理端：内容管理、公告编辑、文章编辑、动作库管理、动作图片上传、饮食模板管理、用户管理和注册审核。
- 前端体验：小蓝鼠纯色品牌标识、暖色页面背景、固定侧边栏、自适应菜单和可拖动背景音乐光碟。
- 后端能力：用户认证、权限判断、训练计划生成、训练周期、动作替换、动作库查询和维护、打卡统计、饮食推荐、饮食模板管理、公告文章管理、H2 本地库和 openGauss 初始化脚本。
- 当前主要风险：前端 `mainpage.vue` 体积过大，页面状态、接口调用和 UI 都集中在一个文件里；旧页面组件仍然保留在 `src`；部分文档内容需要和当前功能保持同步。

## 阶段一：稳定基线

目标：把当前版本整理成一个可继续开发、可交接、可持续迭代的稳定基线。

### 1. 文档同步

任务：

- 更新 README 中的当前功能说明。
- 修正默认账号、动作图片、上传目录等实际状态。
- 补充普通端和管理端的功能边界。
- 增加当前开发启动、验证命令和常见问题说明。

验收标准：

- 新人只看 README 能拉起前后端。
- README 描述和当前页面功能一致。
- H2、openGauss、上传目录、默认账号说明没有互相矛盾。

### 2. 旧代码清理

任务：

- 清理未被路由引用的旧页面组件。
- 保留仍有参考价值的代码前先确认是否已经被新版页面替代。
- 删除无效导入、废弃变量和历史样式。

验收标准：

- `vue-project/src/router/index.js` 引用的页面和实际保留页面一致。
- `npm run build` 通过。
- 删除旧文件不影响普通端和管理端功能。

### 3. 基础验收清单

任务：

- 建立项目验收清单，覆盖登录、注册审核、训练需求、训练计划、动作指导、动作库管理、打卡、复盘、饮食建议、公告文章。
- 每次大改后按清单快速验证。

验收标准：

- 文档中有明确的人工验收路径。
- 普通用户 `demo` 和管理员 `admin` 两种角色都覆盖。

### 4. 当前版本提交

任务：

- 在完成文档和旧代码清理后，提交一个稳定基线版本。
- 提交范围只包含当前产品化基线相关改动，避免混入临时数据库文件、运行日志和本地上传测试文件。

验收标准：

- `git status` 中无未确认的产品代码改动。
- 前端构建和后端测试通过。

## 阶段二：前端结构重构

目标：降低前端维护成本，让后续功能扩展不再继续堆进单个大文件。

### 1. 页面拆分

建议拆分结构：

```text
vue-project/src/
  views/
    LoginView.vue
    UserDashboard.vue
    AdminDashboard.vue
  components/
    AppSidebar.vue
    PageHeader.vue
    MetricCard.vue
    HeatmapGrid.vue
    ArticleCard.vue
    ExerciseGuideCard.vue
  modules/
    training/
    nutrition/
    content/
    admin/
  services/
    userApi.js
    trainingApi.js
    exerciseGuideApi.js
    contentApi.js
```

任务：

- 将 `mainpage.vue` 拆成普通端入口、管理端入口和公共组件。
- 保持现有后端接口名称不变。
- 保持现有功能不减少。
- 保留当前已经修过的响应式布局和弹窗布局规则。

验收标准：

- 普通端和管理端仍然按角色分离。
- 页面切换、详情页路由、弹窗编辑、动作上传都可用。
- 单个 Vue 文件职责清晰，不再把所有页面堆在一个文件内。

### 2. 接口层整理

任务：

- 将当前散落在页面内的 `api.get`、`api.put`、`api.post` 封装到领域 API 文件。
- 统一 `unwrap`、错误提示、请求参数处理。
- 保留 token 使用方式和现有后端兼容。

验收标准：

- 页面组件不直接拼复杂请求参数。
- 接口调用路径集中，后续改接口时不需要全局搜索页面代码。

### 3. UI 状态规范

任务：

- 统一 loading、empty、error、success 状态。
- 表单增加必填校验和合理错误提示。
- 统一列表刷新、弹窗保存、删除确认交互。
- 给长内容、空内容、上传失败、接口失败提供稳定展示。

验收标准：

- 用户不会看到空白页面或无解释的失败。
- 管理端编辑和删除操作反馈明确。
- 窄屏和中等屏宽不出现明显越界、重叠。

## 阶段三：核心产品功能深化

目标：把项目从“能演示的健身系统”升级为“功能逻辑更完整的训练管理产品”。

### 1. 训练计划升级

当前完成：

- 训练计划动作卡片已显示动作推荐理由、今日强度、训练重点和进阶/降级建议。
- 训练计划会从动作库读取肌群、难度、常见错误、禁忌、适合人群和替代动作。
- 训练计划动作勾选结果已保存到 `plantaskrecords`，刷新页面后仍能保留当天完成状态。
- 训练计划页已展示本周完成率、连续打卡、恢复提示和下一步调整建议。
- 现有接口名称保持不变：`/fitnessplan/getsplitmode`、`/fitnessplan/getactiontask`。

当前继续完成：

- 已增加训练周期接口 `/fitnessplan/cycle`，返回当前周次、阶段、周期目标、恢复日和周期时间。
- 已增加计划调整记录接口 `/fitnessplan/adjustments`，训练计划洞察中同步返回最近调整记录。
- 已增加动作替换接口 `/fitnessplan/action/replace` 和动作卡片内的可替换动作按钮，优先选择同动作模式、不同器材的动作。

建议新增数据方向：

- 用户训练周期表 `trainingcycles`：记录当前周期、目标、开始日期、结束日期。
- 动作完成记录表：已先用 `plantaskrecords` 记录每个动作是否完成、组数、次数、主观难度。
- 计划调整记录表 `planadjustmentrecords`：记录系统为什么建议加量、降量或恢复。
- 动作替换关系表 `exercisealternatives`：记录动作之间的替换关系、原因和优先级。

验收标准：

- 训练计划不只是展示固定动作，而能根据用户记录产生调整建议。
- 用户能看懂今天练什么、为什么这么练、完成后下一步是什么。

### 2. 动作指导库升级

当前完成：

- 数据表 `exerciseguides` 已扩展：主肌群、辅助肌群、难度、禁忌、常见错误、适合人群、替代动作。
- 当前内置动作已按动作名称和器材逐条补齐上述字段，不再只按动作模式套用同一套内容。
- 管理端新增/编辑动作弹窗已支持维护上述字段。
- 普通用户动作指导详情已展示难度、肌群、禁忌、常见错误、适合人群和替代动作。
- 后端会对旧数据做默认内容兜底，避免新字段为空导致动作详情缺核心信息。
- 管理端已支持按缺图片、缺步骤、缺要点、缺常见错误、缺替代动作和整体不完整动作筛选。
- 动作库接口会返回内容完整度评分，管理端列表可直接看到待补质量状态。
- 已新增 `exercisealternatives` 表和同动作模式替换接口，训练计划卡片可直接选择可替换动作。

后续任务：

- 继续扩大替代动作关系数据覆盖范围。
- 后续可按动作风险、器材可用性和用户水平做更细替换排序。

建议新增数据方向：

- 动作肌群字段：主肌群、辅助肌群。
- 动作难度字段：新手、进阶、熟练。
- 动作质量状态字段：是否有图片、是否有步骤、是否有要点、是否完整。
- 替代动作关系表：同模式、同目标部位、不同器材之间的替代关系。

验收标准：

- 动作库可以支撑训练计划生成，而不是只做静态百科。
- 管理员能快速找出不完整动作并补齐内容。
- 普通用户进入动作详情后能明确知道动作怎么做、容易错在哪。

### 3. 饮食推荐细化

当前完成：

- 后端兼容保留 `nutritionpreferences` 表和 `/nutrition/preference` 接口，普通用户端当前不再展示偏好维护表单。
- 饮食建议已根据训练目标、训练频率、餐单模板、食材替换、外食场景和执行提醒生成推荐内容。
- 前端饮食建议页已去掉普通用户侧偏好设置表单，减少饮食页操作负担。
- 前端饮食建议页已去掉推荐历史，避免刷新建议时堆积重复记录。
- 已新增 `fooditems`、`mealtemplates`、`foodreplacements`、`eatingscenarios` 表。
- 饮食推荐已读取食材、餐单模板、替换规则和食堂/便利店/外卖/家庭做饭场景建议。
- 管理端已新增饮食模板维护入口，可维护食材、餐单模板、替换规则和外食场景。

后续任务：

- 继续扩充不同训练目标、用餐场景和预算层级下的餐单模板。
- 后续可加入每日摄入记录，让推荐根据真实饮食执行情况调整。

建议新增数据方向：

- 兼容偏好表 `nutritionpreferences`：保留历史用户饮食偏好字段，当前不作为普通用户侧主要操作入口。
- 饮食模板表 `mealtemplates`：按目标、餐次和场景保存餐食模板。
- 食材库表 `fooditems`：记录食材类别、宏量营养和适用场景。
- 替换规则表 `foodreplacements`：记录同类食物替换关系。
- 饮食场景表 `eatingscenarios`：记录食堂、便利店、外卖、家庭做饭策略。

验收标准：

- 饮食建议不再只是数字，而是能落到餐食结构和可执行食材。
- 用户能根据训练目标、训练频率和后台模板库得到不同推荐。
- 推荐结果能解释来源，避免看起来像硬编码文案。

### 4. 内容体验升级

任务：

- 文章增加封面、摘要、标签、阅读时长。
- 公告区和文章区视觉区分更明显。
- 普通用户只阅读内容，管理员只在管理端维护内容。
- 文章详情支持长文阅读体验：目录、段落间距、返回列表、相关推荐。

验收标准：

- 普通用户不会看到编辑入口。
- 管理员维护内容不影响普通用户阅读路径。
- 长文页面阅读体验稳定，不出现拥挤或过长不可控的问题。

## 推荐执行顺序

建议按以下顺序推进：

1. 更新 README 和补充验收清单。
2. 清理旧页面组件。
3. 拆分 `mainpage.vue`，完成前端结构重构。
4. 封装前端 API 层。
5. 升级动作指导库数据结构和管理页面。
6. 升级训练计划生成和动作替换逻辑。
7. 细化饮食推荐。
8. 优化内容阅读体验。

优先级判断：

- 第一优先级：文档同步、旧代码清理、前端拆分。
- 第二优先级：动作库升级，因为训练计划依赖动作库质量。
- 第三优先级：训练计划动态调整。
- 第四优先级：饮食推荐细化和内容体验优化。

## 暂不纳入本阶段的事项

以下内容暂不作为当前产品化阶段的目标：

- Docker Compose。
- CI/CD。
- 线上部署流程。
- 生产监控和日志平台。
- 多环境配置中心。
- 云存储和 CDN。

这些内容在产品功能稳定后再统一规划。
